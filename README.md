## 📌 Project Overview

This project demonstrates the implementation of a **Security Operations Center (SOC) Homelab** designed to simulate real-world cyber attacks within a fully isolated virtual environment. The objective is to understand how enterprise Security Information and Event Management (SIEM) solutions collect, normalize, correlate, and analyze security telemetry generated from endpoints.

The environment consists of three virtual machines connected through an isolated NAT Network:

- **Kali Linux** acting as the attacker machine.
- **Windows 10 Pro** acting as the monitored endpoint.
- **Ubuntu Server** hosting the Wazuh Manager, Indexer, and Dashboard.

To improve detection quality, **Microsoft Sysmon** was deployed on the Windows endpoint with a customized configuration based on the SwiftOnSecurity template, while **Wazuh Agent** was manually configured to collect Sysmon and Windows Defender telemetry.

The lab simulates common attack scenarios such as:

- SMB Enumeration & Brute Force
- Malware Execution Simulation (EICAR)

Each attack is validated through the Wazuh SIEM dashboard to verify that endpoint telemetry is successfully collected, parsed, and transformed into actionable security alerts.

# 📡 Telemetry Engineering

## Sysmon Configuration

To enhance endpoint visibility, Microsoft Sysmon was installed and configured using a customized version of the **SwiftOnSecurity** configuration template.

The customized configuration is available in:

```
sysmonconfig-export.xml
```

The configuration emphasizes high-value telemetry while reducing unnecessary noise generated by trusted services.

Key monitored events include:

- Process Creation
- Network Connection
- DNS Query
- Image Load
- Driver Load
- File Creation
- Registry Modification

DNS Query filtering (Event ID 22) was customized to suppress common trusted domains such as:

- Google
- Microsoft
- CDN Providers

This allows analysts to focus on suspicious outbound communications.

Telemetry validation can be observed in:

- **`sysmon-events-log.png`**
- **`sysmon-events-detail.png`**

## Wazuh Agent Configuration

The Windows endpoint was connected to the SIEM infrastructure using the Wazuh Agent.

The configuration file is available in:

```
wazuh-agent.conf
```

Manual `<localfile>` entries were added to collect logs from:

- Microsoft-Windows-Sysmon/Operational
- Windows Defender

This configuration ensures that endpoint telemetry is forwarded continuously to the Wazuh Manager for correlation and analysis.

# 🛠️ Custom Rules

In addition to Wazuh's built-in detection capabilities, this project includes custom detection rules developed to improve detection accuracy and identify attack behaviors specific to the homelab environment. These rules were implemented in the Wazuh Manager through the local ruleset (`/var/ossec/etc/rules/local_rules.xml`).

The implementation of these custom rules is shown in **`custom-rules.png`**, which contains the detection logic used to identify SMB brute-force activity and PowerShell-based EICAR file creation.

---

## 📜 Custom Detection Rules

| Rule ID | Severity | Detection | Log Source | MITRE ATT&CK |
|-----------|----------|------------------------------|----------------------------|------------------------------|
| **100001** | High (10) | SMB Brute Force via NetExec | Windows Security (Event ID 4625) | T1110.003 – Password Spraying |
| **100002** | Critical (12) | PowerShell EICAR File Drop | Sysmon (Event ID 11) | T1059.001 – PowerShell |

The corresponding Wazuh rule definitions can be viewed in **`custom-rules.png`**, illustrating how both detection rules were implemented within the Wazuh Manager.

---

## 🔍 Detection Strategy

### Rule 100001 — SMB Brute Force Detection

This rule correlates multiple Windows authentication failures (Event ID 4625) within a configurable time window to identify potential SMB brute-force attacks. Instead of generating an alert for every failed authentication, the rule consolidates repeated events into a single high-severity alert, reducing alert fatigue while preserving attack visibility.

### Rule 100002 — PowerShell File Creation Detection

This rule monitors **Sysmon Event ID 11 (File Creation)** to detect PowerShell creating files containing the EICAR test signature. By focusing on file creation behavior rather than solely monitoring process creation events, the rule provides more reliable detection of PowerShell-based activity.

---

## 🧪 Rule Validation

Each custom rule was validated using the **Wazuh Logtest** utility before deployment to verify proper rule matching and alert generation. After validation, the rules were added to the local Wazuh ruleset, as shown in **`custom-rules.png`**, and used throughout the attack simulations presented in this project.

# 📚 Key Takeaways

This project strengthened my understanding of how enterprise SOC environments collect and analyze endpoint telemetry. Rather than relying solely on default Windows Event Logs, integrating Sysmon significantly improves endpoint visibility by generating detailed security-relevant events while minimizing unnecessary noise through custom filtering.

Additionally, manually configuring the Wazuh Agent to ingest both Sysmon and Windows Defender logs demonstrated how multiple telemetry sources can be centralized within a SIEM platform to provide richer context for investigations.

Through attack simulations involving SMB brute-force attempts and EICAR malware execution, I gained hands-on experience validating detection pipelines, analyzing security alerts, and understanding how endpoint activities are transformed into actionable intelligence for Blue Team operations.
